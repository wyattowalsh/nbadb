# Export and Share Artifact Surfaces

This note maps the repo's user-facing artifact surfaces for getting results out of the chat app or the docs site. It is about analysis artifacts and sharing surfaces, not the warehouse pipeline's SQLite/DuckDB/Parquet/CSV distribution lane.

## Surface map
| Surface | Primary home | What it produces | Best use |
| --- | --- | --- | --- |
| Query-result downloads | chat UI | CSV, XLSX, JSON files from tabular results | quick export of the last query result |
| Spreadsheet export | chat UI and chat sandbox helpers | self-contained HTML spreadsheet with editable grid | light in-browser editing before sharing or saving |
| Programmatic table export | chat sandbox helpers | CSV, XLSX, JSON | reproducible analyst-directed exports from Python |
| Embed export | chat sandbox helpers | self-contained HTML snippet for Plotly figures | drop a chart into a blog or site |
| Social card export | chat sandbox helpers | branded 1200x630 PNG | one-off analyst share asset |
| Thread export | chat sandbox helpers | plain-text numbered thread | social post drafting |
| Docs playground CSV copy | docs MDX playground | CSV copied to clipboard | moving small result sets out of the browser sandbox |
| Docs OG images | docs app routes | site-level and page-level Open Graph images | link previews and social unfurls |

## Chat lane

### Query-result actions are the main no-code export surface
When a chat step returns a table, Chainlit stores a lightweight export payload in session state and adds action buttons for:
- `CSV`
- `XLSX`
- `JSON`
- `Edit as Spreadsheet`

Those actions rebuild a DataFrame from the stored payload and return inline downloadable files. The payload is capped to 100 rows, so this surface is for convenient result export, not bulk delivery.

### Python helpers expose the full artifact lane
The sandbox MCP advertises the artifact helpers directly in the `run_python` contract:
- `to_csv(df, name)`
- `to_xlsx(df, name)`
- `to_json(df, name)`
- `export(df, name, fmt)`
- `to_spreadsheet(df, name)`
- `to_embed(fig, title)`
- `to_social(df_or_fig, headline, subtitle)`
- `to_thread(insights)`

This is the richer export lane because it works after arbitrary Python post-processing, not just raw SQL-table output.

### Spreadsheet export is HTML-first
Both the chat action callback and the sandbox helper route spreadsheet generation through the shared AG Grid template in `chat/server/_spreadsheet_template.py`.

Current repo reality:
- the spreadsheet artifact is a downloadable `.html` file
- it supports in-browser editing, sorting, filtering, and reset
- the toolbar currently exports `CSV` and `JSON`

One helper docstring in `chat/server/_preamble.py` still says the spreadsheet includes XLSX export, but the shared template currently implements CSV and JSON only. Treat the template as the canonical runtime behavior.

### Embeds and social cards are chat-owned share artifacts
- `to_embed(fig, title)` serializes a Plotly figure into a portable HTML snippet wrapped in an `nbadb-embed` container.
- `to_social(...)` renders a branded `1200x630` PNG card named `social_card.png`.
- `to_thread(...)` emits a numbered text thread.

These are explicit outbound artifact surfaces for user-directed sharing. They are distinct from docs-side metadata images.

## Docs lane

### Docs does not own the main analyst export flow
The docs app does not expose the same CSV/XLSX/JSON/spreadsheet artifact set as chat. Its closest tabular export surface is the MDX SQL playground's `CSV` copy action, which copies the visible result set to the clipboard instead of generating a downloadable file.

That makes docs useful for demo, teaching, and small-sample extraction, but not the primary share/export lane for analyst outputs.

### Docs owns OG image generation
The docs app has two Open Graph image surfaces:
- `docs/app/opengraph-image.tsx` for the site-wide social card
- `docs/app/docs-og/{catch-all}/route.tsx` for per-page OG images based on section and page metadata

`docs/app/layout.tsx` wires `/opengraph-image` into both `openGraph.images` and `twitter.images`, so the docs site automatically produces share previews for links.

### Docs images are metadata assets, not exported analysis artifacts
Docs OG images serve page discovery and social unfurling. They are generated by routes and metadata, not by an analyst asking for a downloadable artifact.

That is the key split:
- chat produces user-directed exports
- docs produces route-directed share metadata

## Placement rules
- Use chat query actions for fast table export of the last result.
- Use chat sandbox helpers when the export must follow Python analysis or custom formatting.
- Use `to_spreadsheet(...)` when the recipient needs an editable browser artifact.
- Use `to_embed(...)` for portable chart embeds.
- Use `to_social(...)` for one-off analyst share cards.
- Use docs OG routes for site and page preview images only.
- Treat docs playground CSV as a convenience copy surface, not a full export system.
