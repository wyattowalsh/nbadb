### /docs/diagrams/er-diagram (authored)

**Status**: Good curated page with one moderate-size ER diagram.

- Reading guide ("How to Read the Board") with scan-order table is helpful.
- Schema category summary table (17 dim, 72 fact, 2 bridge, 16 derived, 4 views).
- Cross-links to er-auto, Schema Reference, Table Lineage.

- **Defect D-01**: The `fact_game_log` entity in the ER diagram references `dim_game` and `dim_season` but `dim_season` is not defined as an entity with attributes in the diagram block. It appears in relationships but not as a full entity definition. Similarly, `dim_lineup`, `dim_draft`, and `dim_play_type` lack attribute definitions.

### /docs/diagrams/er-auto (auto-generated)

**Status**: Functional but with performance concerns.

- Single `erDiagram` block spanning ~2,340 lines of Mermaid source.
- Approximately 109 entity definitions, each with full column listings including types.
- No relationships defined between entities -- only entity attribute blocks.
- **Defect D-02 (P1)**: The auto-generated ER diagram includes internal Pandera implementation details as fields. Every entity has `type[pandera.api.polars.model_config.BaseConfig] Config` as its first field. This is schema metadata noise, not a real column. Mixin classes like `__team_dashboard_fantasy_metrics_mixin` also appear as entities, which are implementation artifacts.
- **Defect D-03 (P1)**: No relationships are defined in the auto-generated diagram. An ER diagram with 109 entities and zero relationships provides no relational context -- it is effectively a column inventory, not an ER diagram.
- **Defect D-04 (P2)**: Performance risk. A 2,340-line Mermaid ER diagram with 109 entities will be slow to render client-side. Mermaid's ER layout algorithm scales poorly beyond ~30 entities. Expected render time: 5-15+ seconds depending on device. No timeout or chunking strategy exists.
- **Missing**: No "generated by" surface or timestamp indicating when the diagram was last regenerated.

## Mermaid Rendering Quality

### Theme awareness
- The `getMermaidTheme()` function in `mermaid.tsx` reads 20+ CSS custom properties (`--card`, `--primary`, `--border`, etc.) to build a Mermaid `base` theme. This ensures diagrams adapt to light/dark mode.
- **Caveat**: The pipeline-flow diagram uses inline `style` directives with hardcoded hex colors that will override theme variables. In dark mode, the light fill colors and dark text may still be legible, but the pastel fills will not invert.

### Caching
- A `Map<string, Promise<unknown>>` cache keyed by `${chart}-${resolvedTheme}` avoids re-rendering the same diagram. This is effective for theme switching but does not help with initial render of large diagrams.

### Rendering pipeline
- `Suspense` wraps the render with a `MermaidFallback` that shows the first 12 lines of Mermaid source as a preview. Good UX during load.
- The `mermaid` library is lazy-loaded via `import("mermaid")` with its own cache.

### Missing: Error boundary
- **Defect D-08 (P2)**: If `mermaid.render()` throws (syntax error, timeout, memory), the `use()` hook in `MermaidContent` will throw to the nearest `Suspense` boundary. Since `Suspense` catches only pending promises (not errors), the error will propagate up to React's default error handling, which shows a blank white screen or the default Next.js error page. No `ErrorBoundary` exists anywhere in the component tree.

## Zoom/Pan UX

### Implementation quality
- `useZoomPan` hook uses `useReducer` with four actions: `ZOOM`, `PAN`, `SET`, `RESET`.
- Scale range: 0.1x to 20x (generous range).
- Scroll-to-zoom uses non-passive wheel listener with `preventDefault()` -- correct approach.
- Pointer-based drag panning with `setPointerCapture` -- correct approach.
- `fitToView()` calculates scale to fit content within viewport, capped at 1x.
- `willChange: "transform"` promotes the canvas to its own compositor layer -- good for performance.

### Keyboard controls
- `+`/`=`: zoom in
- `-`: zoom out
- `0`/`Home`: fit to view
- Arrow keys: pan (50px step)
- Double-click: reset view
- **Observation**: Good keyboard accessibility. The viewport has `tabIndex={0}`, `role="application"`, and `aria-label`.

### Zoom controls UI
- Three buttons (zoom out, zoom in, fit to view) with Lucide icons.
- Percentage display with `aria-live="polite"`.
- `role="toolbar"` on the control group.

### Missing capabilities
- **Defect D-09 (P2)**: No pinch-to-zoom support. The `useZoomPan` hook only handles pointer events (mouse/touch single-point) and wheel events. Multi-touch pinch gestures are not implemented. The CSS `touch-action: none` on the viewport disables the browser's native zoom, so mobile users have no zoom mechanism at all beyond the button controls.
- **Missing**: No minimap or overview indicator for large diagrams. When zoomed into a 300-node graph, users have no sense of where they are in the overall layout.

## Enhancement Ideas

1. **Use the LineageExplorer component (D-10 fix)**: The interactive three-panel lineage explorer (`lineage-explorer.tsx`) is fully built with search, layer filtering, depth controls, and upstream/downstream BFS traversal. It reads from `lineage.json`. Add a `<LineageExplorer data={...} />` call to either the lineage index or lineage-auto page to surface this work. This would largely solve D-05 and D-07 as well by providing an interactive alternative to the massive Mermaid graph.

2. **Split or replace large auto-generated diagrams (D-04, D-05)**: For er-auto, consider either (a) splitting the single diagram into per-layer diagrams (one for dims, one for facts, one for aggs), or (b) replacing the Mermaid diagram with the existing `SchemaExplorer` interactive component. For lineage-auto, replace the Mermaid diagram with `LineageExplorer`.

3. **Fix the autogen script (D-02, D-03)**: Filter out `Config` fields and mixin/internal classes from the ER diagram output. Add relationship edges based on `fk_ref` metadata already present in star schemas.

4. **Add ErrorBoundary (D-08)**: Wrap the `<MermaidContent>` Suspense in an ErrorBoundary that falls back to the `MermaidFallback` component with an error message. This prevents full-page crashes on render failure.

5. **Add pinch-to-zoom (D-09)**: Extend `useZoomPan` to track two-pointer gestures and compute pinch distance delta for scale changes. This is the standard pattern for touch-friendly zoom.

6. **Add generation metadata (D-06)**: Include a small info surface on auto-generated pages showing last-regenerated timestamp and the command to regenerate.

7. **Add a visual diagram to Endpoint Map**: A Sankey or grouped flowchart mapping endpoint families to target table types would strengthen the "Diagrams" section identity of this page.

8. **Add render timeout**: For large diagrams, add a `Promise.race` with a 10-second timeout around `mermaid.render()`. If it times out, show the fallback with a message suggesting the user try a smaller view.

## Notes

- The Mermaid component architecture is well-designed: lazy loading, theme-aware rendering, Suspense fallback with source preview, zoom/pan with keyboard accessibility, and render caching.
- The zoom/pan hook is production-quality for mouse/desktop usage. The main gap is mobile touch.
- The curated pages (er-diagram, pipeline-flow, table-lineage, column-lineage) are excellent -- well-written reading guides, appropriate diagram sizes, and clear cross-linking.
- The auto-generated pages are the weak point: too large for Mermaid, missing metadata, and in the case of er-auto, including implementation artifacts.
- The biggest missed opportunity is the unused `LineageExplorer` component, which is a fully functional interactive lineage browser that was built but never wired into any page.
- Browser-based visual verification was not possible (Chrome DevTools connection unavailable). The performance assessments for large diagrams are based on known Mermaid rendering characteristics at the stated node/edge counts.
