# Troubleshooting Playbook

Use this page as the **film room after a broken possession**. Start with the artifact or command named in the failure, then work the shortest route back to a clean rerun.

## Dead-ball triage

| If the failure mentions…                          | Start here                        |
| ------------------------------------------------- | --------------------------------- |
| Missing environment, lockfile, or package surface | Dependency inventory              |
| Empty tables or failing checks                    | Data quality                      |
| Endpoint coverage drift                           | Extract completeness              |
| Full Extraction lane failures                     | Full Extraction control plane     |
| Generated docs or docs build drift                | Docs-autogen artifacts            |
| Download/upload or publish misses                 | Kaggle and common CLI/CI failures |

<Callout type="info">
  Do not debug the whole pipeline by default. Start from the exact artifact,
  report, or command named in the error message and only widen the search if
  that lane stays cold.
</Callout>

## Use the narrowest recovery loop

1. Regenerate or reopen the exact artifact named in the failure.
2. Inspect the JSON or Markdown output, not just the terminal text.
3. Fix the local cause first.
4. Rerun the narrowest failing command before you escalate to a wider pipeline command.

## Dependency inventory artifact (`dependency-inventory.json`)

**Best for:** missing lockfiles, missing docs package paths, or suspicious environment surface.

### 1) Generate

```bash
uv run python -m nbadb.core.dependency_inventory --project-root . --output artifacts/health/local/dependency-inventory.json
```

### 2) Diagnose

```bash
python - <<'PY'
import json

data = json.load(open("artifacts/health/local/dependency-inventory.json"))
print("summary:", data["summary"])
print("lockfile_present:", data["lockfile"]["present"])
print("pyproject_present:", data["pyproject"]["present"])
print("docs_package_present:", data["docs_package"]["present"])
PY
```

### 3) Remediate

- If `lockfile_present` is `false`, run `uv sync` (or provide `--lock-path`) and commit the lockfile used by CI.
- If `docs_package_present` is `false`, run from repo root or pass `--docs-package-path /abs/path/docs/package.json`.
- If `summary.package_count` is unexpectedly low, check `--project-root` and rerun with an absolute path.

## Data-quality JSON artifact (`data-quality-report.json`)

**Best for:** empty tables, failing checks, or “database not found” quality runs.

### 1) Generate

```bash
nbadb scan --report-path artifacts/health/local/data-quality-report.json
```

### 2) Diagnose

```bash
python - <<'PY'
import json

report = json.load(open("artifacts/health/local/data-quality-report.json"))
print("summary:", report["summary"])
print("failed:", [r["message"] for r in report["results"] if not r["passed"]][:10])
PY
```

### 3) Remediate

- If you see `Error: database not found. Run 'nbadb init' first.`, seed the data directory first with `nbadb init` or `nbadb download` plus a refresh command.
- If `summary.total` is `0`, verify the DuckDB file exists at the configured `--data-dir`.
- If checks fail, inspect the reported tables/columns and rerun the refresh command that matches the scope you need (`daily`, `monthly`, or `backfill run`).

**Tip:** `nbadb scan` supports `--fail-on error` for strict CI enforcement and `--category` to focus on specific check families (e.g. `data_quality`, `missing_table`).

## Endpoint coverage artifacts (`artifacts/endpoint-coverage/*`)

**Best for:** extractor/staging drift and runtime coverage gaps.

### 1) Generate

```bash
# Local runtime coverage audit
uv run nbadb extract-completeness

# Runtime source coverage gate
uv run nbadb extract-completeness --require-full

# Full upstream docs/tools + runtime source coverage gate
uv run nbadb extract-completeness --require-full --endpoint-analysis-docs-root /path/to/nba_api

# Full upstream docs/tools + runtime + model-contract gate
uv run nbadb extract-completeness --require-full --require-model-contract --endpoint-analysis-docs-root /path/to/nba_api
```

By default this writes:

- `artifacts/endpoint-coverage/endpoint-coverage-matrix.json`
- `artifacts/endpoint-coverage/endpoint-coverage-summary.json`
- `artifacts/endpoint-coverage/endpoint-coverage-report.md`

Use `--output-dir` to write artifacts elsewhere.

The summary distinguishes source coverage from model ownership. In practice that lets you separate "we are not extracting or staging this surface" from "we land it, but we still need a downstream transform or an explicit exclusion." It also reports `star_schema_coverage`, which answers a different question: "the transform exists, but do we have a schema-backed final-tier contract for its output?"

### 2) Diagnose

```bash
python - <<'PY'
import json

summary = json.load(open("artifacts/endpoint-coverage/endpoint-coverage-summary.json"))
print("runtime:", summary["runtime_version"], "classes:", summary["runtime_endpoint_class_count"])
print("coverage:", summary["coverage"])
print("star_schema_coverage:", summary.get("star_schema_coverage", {}))
PY
```

### 3) Remediate

- If `runtime_version` is `unknown` or the class count is `0`, make sure `nba_api` is installed in the active environment.
- If `runtime_gap` is high, validate runtime references in `src/nbadb/extract/stats/*.py` and endpoint names in `src/nbadb/orchestrate/staging_map.py`.
- If `extractor_only` rows appear, either add the endpoint to the staging map or normalize the coverage aliasing.
- If `schema_missing_transform_outputs` is non-zero, the pipeline is producing transform outputs without a matching `nbadb.schemas.star` contract. Treat that as a data-modeling gap, not an extractor/staging gap.

## Full Extraction control plane

**Best for:** GitHub Actions lane recovery, chained manifests, and distinguishing resumable work from real pipeline failures.

### 1) Read the lane outcomes

The full extraction workflow reduces each lane to one final outcome in `lane-metadata.json`:

| Outcome            | Meaning                                                                                         | Next action                                                          |
| ------------------ | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| `complete`         | The lane extracted successfully.                                                                | Preserve it as `resume_only` in the next manifest.                   |
| `needs_resume`     | The lane timed out or stopped after persisted DuckDB, journal, or staging progress.             | Keep it active and reuse cache or durable lane state.                |
| `contract_blocked` | The endpoint scope is documented as unsupported by upstream/runtime evidence.                   | Skip retry, keep it in `extraction-audit.json`.                      |
| `pipeline_failure` | Classified extraction, VPN, runner, artifact, manifest/control-plane, or secret-safety failure. | Retry only within the failure-class budget or fix before redispatch. |

`vpn_network_error`, `vpn_auth_failure`, and `vpn_connect_timeout` are all
`vpn_egress` failures with a bounded retry budget. They must not be collapsed into
the one-attempt application-failure cap.

### 2) Inspect the chain artifact

```bash
python - <<'PY'
import json
from pathlib import Path

root = Path("artifacts/full-extraction")
print(json.load(open(root / "resume-summary.json")))
print(json.load(open(root / "extraction-audit.json"))["outcome_counts"])
PY
```

### 3) Understand scheduling, discovery, and checkpoints

Fresh GitHub Actions full-extraction runs default to `standard`. It preserves the
same coverage fingerprint as the smaller profiles while reducing runner setup,
VPN connection, artifact, and checkpoint overhead. Durable incomplete-lane state
makes these broader shards resume-safe. Use `balanced-small` after repeated
endpoint timeouts, and reserve `micro` for isolating a provider or contract failure
to one-season chunks.

The scheduler divides active work into fresh, partial-progress, retry, and
infrastructure queues. When every queue has work, each ten-slot rotation targets
five fresh lanes, three partial-progress lanes, one retry lane, and one
infrastructure lane. The priority score incorporates persisted progress,
throughput tier, endpoint family, estimated cost, and older historical scope. A
family uses a ceiling-rounded 25% cap while another family is available, so small
waves may exceed exactly 25% and diversity can override the preferred queue. Inspect
`scheduler_diagnostics.queue_counts`, the cost summaries, and `top_cost_lanes` in
the manifest when the first wave is surprising.

Manifest generation also enforces parameter-level executability. The schema-backed
`player_vs_player`, `team_vs_player`, `team_and_players_vs`, and the extractor-only
`team_and_players_vs_players` alias are currently `contract_not_modeled_yet` for
historical fan-out: the durable
player-team-season workload contains affiliations, not observed comparison pairs or
two opposing lineups. Creating a Cartesian product would be both quadratic and
semantically unproven. The three canonical families remain visible in coverage
artifacts with revalidation paths; the extractor-only alias is represented by the
canonical `team_and_players_vs` staging windows and its own extraction exclusion.
All four concrete names are omitted from fresh manifests and cause an old explicit
manifest to fail during planning before network preflight.

`league_game_log` is a first-class discovery-seed output and is therefore absent
from the extract matrix. Canonical coverage rows can describe multiple alternate
wrappers, so manifest generation resolves each support window's staging key back to
its concrete runtime endpoint and parameter pattern. For example, both
`player_game_logs`/`season` and `player_game_logs_v2`/`player_season` are scheduled;
the canonical family name is never substituted for a different concrete route.

Before lane fan-out, one `discovery_seed` job derives the exact season and
season-type combinations required by the current `github_matrix.include`. It
centralizes game/date, player-ID, and player-team-season discovery behind one VPN
tunnel when VPN mode is effective. Seasons are grouped only when they share the
same season-type set, so the seed does not invent a broader cross product.

VPN acceptance is endpoint-aware. The action first proves the full tunnel route
and a changed public IP, confirms that GitHub remains reachable through the tunnel,
validates a non-empty `TeamYears` result set, then runs bounded `common_all_players`
and `league_game_log` canaries through the installed `nbadb`/`nba_api` extraction
stack. The player canary requires both IDs and at least one row with positive
player/team membership. Inspect
`nba-probe-status`, `nba-probe-diagnostic`, `attempted-servers-json`, and
`failed-servers-json`. Preflight also emits `auth-source`; discovery and extract
jobs reuse the source that actually passed instead of replacing proven configured
credentials after a transient lane rejection. A server that passes a generic IP
probe but times out or returns malformed NBA content is disconnected and added to
the failed-server list before the next recommendation is tried. Runtime failures are
excluded from every fallback technology. Preflight failures feed discovery and the
capacity gate. A later fail-closed aggregation job validates every successful capacity
marker and merges preflight and discovery failures with those probes' failed hosts. A
capacity probe that does not complete remains diagnostics-only and blocks the matrix;
its unvalidated host inventory is not promoted into extraction or a child manifest.

Preflight and discovery also expose the servers that pass every NBA probe. The serial
discovery job tries the disconnected preflight-proven host before fresh recommendations.
Extract lanes receive both successful hosts as a verified pool before consulting fresh recommendations.
Logical lane index `N` may use only preferred host `N`; preferred assignment never
wraps modulo when the lane count exceeds the verified-host count. Later lanes use
disjoint fresh recommendations instead of falling through to another active lane's host.
This avoids rediscovering known-good exits without collapsing parallel runners onto
one NordVPN server. Quarantine and per-protocol attempt ceilings still take precedence.

The connector fetches and caches a wider Nord-recommended country pool than it
attempts for each protocol. It first builds one stable city/station-network-diverse
ordering, then assigns adjacent lane or rerun indexes to disjoint pool segments while
pool capacity permits. This keeps current Nord health recommendations while preventing one cluster of NBA-blocked
exits from consuming every fallback slot. If exclusions underfill an assigned segment,
the cached recommendation snapshot expands additively and fills only from that
selector's reserve segment, without rebasing other selectors. `SERVER_LIMIT` is the
hard per-protocol attempt ceiling across all recovery rounds; `SERVER_POOL_SIZE`
improves candidate diversity without increasing attempts.

An OpenVPN `AUTH_FAILED` response is not enough to classify a server as unhealthy:
the connector does not add that host to the chain quarantine. After discovery,
configured credentials must first pass a concurrent capacity gate sized to the actual
active lane count and `vpn_parallelism`. Each connected probe publishes a run-attempt
marker, waits for every peer marker while keeping its tunnel open, and rechecks its
process, route, and exit IP after the barrier. Any probe, marker, barrier, or recheck
failure blocks the entire extraction matrix. Before fan-out, the workflow downloads
every successful `vpn-capacity-connected-run-<run_id>-attempt-<attempt>-lane-<index>` artifact,
validates its exact provenance and tunnel evidence, and publishes
`vpn-quarantine-<chain_id>-run-<run_id>-attempt-<attempt>`. Each extraction connector also reserves one complete initial attempt and bounded
cleanup, one five-minute cooldown, and a complete follow-up attempt and cleanup before it starts an
authentication-capacity probe. It stops at the first rejection, waits, rotates the
lane-specific starting server and preferred protocol, and permits up to three further
authentication rejections in the recovery sweep inside a 12-minute connector budget.
Each server attempt stops before a 120-second internal finalization reserve for OpenVPN
cleanup, work-directory permissions, and action outputs; the supervisor retains a
separate emergency margin.
Preflight and discovery retain their shorter gates. HTTP 401/403 responses fail
immediately instead of being retried as transport faults. If preflight succeeds only
with token-derived credentials, matrix execution is limited to one simultaneous job
without shrinking the planned batch and skips the configured-credential capacity gate.

On chained runs, the seed job downloads
`full-extraction-discovery-artifacts-<chain_id>` from the exact
`lane_manifest_run_id`, adds the current wave's missing scopes, and republishes the
cumulative artifact. If that prior artifact is unavailable, the job reseeds from
scratch and still applies the same exact-scope gate. Extract lanes install the
result after restoring lane state, preventing an older cached database from
replacing current discovery evidence. A required scope counts as covered only
when its exact artifact exists; explicit zero-row game combinations and
player/team pairs are valid coverage, but missing combinations are not. Any
unresolved player season, game combination, or player/team pair increments
`failure_count`, exits the seed command non-zero, and prevents the extraction
matrix from starting. Active-season player IDs, game/type combinations, and
player-team/type pairs are always rediscovered; historical scopes remain reusable.
An aggregate-only player wave still performs one targeted active-season refresh.
Sparse missing/invalid player-team pairs are grouped by identical type sets, so
live fallback does not refetch the surrounding Cartesian rectangle.

The seed gives every request a hard timeout and uses one three-attempt bounded canary
per observed transport or response-shape failure class instead of retrying hundreds
of scopes after a broad outage. Individual discovery calls spend the same bounded
retry budget on transport-transient failures and response-contract or validation
failures, including failures classified through a wrapped cause. A response that
cannot satisfy the required discovery columns is therefore retried rather than being
misclassified as a permanent application error. It writes an atomic schema-v2 summary before
network work and after each completed result. VPN discovery profiles separately cap
in-flight calls and the shared request rate; the default moderate profile uses three
in-flight calls at two requests per second so one accepted exit is not immediately
overloaded by the historical seed. Its 90-minute soft deadline is backed by
a 95-minute process watchdog, reserving upload time before the 120-minute job cap even
when a dependency ignores cancellation. Canonical
`full-extraction-discovery-artifacts-<chain_id>` uploads are success-only and
contain atomic manifest pointers to immutable content-addressed discovery and workload
Parquet generations; each pointer binds exact scope/pairs, schema, row counts, sentinel
semantics, and SHA-256. The schema-v2 summary binds the exact lane-manifest digest. A
standalone verifier independently derives required units from that manifest and reloads
every requested player scope, game combination, and workload generation before upload
and after each lane installs the downloaded bundle.
Incomplete work is retained separately as
`full-extraction-discovery-recovery-<chain_id>-run-<run_id>-attempt-<attempt>`.
A later seed first restores the latest recovery bundle from the current run's prior
attempt, then falls back to the exact manifest source run. It reuses only artifacts
that pass their integrity/schema checks; malformed partial workload pointers are
discarded and reseeded, while malformed canonical bundles fail closed. Valid exact
single-season v1 player/game artifacts are promoted to the current format; aggregate
v1 artifacts are rejected. Recovery state never becomes canonical until the current
seed proves complete coverage and the bundle verifier passes. If the seed fails,
`lane_control` is skipped: no missing lane metadata is synthesized, no retry counters
advance, and no child iteration is dispatched.

Each iteration builds a cumulative checkpoint artifact named
`full-extraction-checkpoint-<chain_id>-iter-<iteration>`. The next manifest carries
that checkpoint pointer forward, so redispatched runs can merge the previous
checkpoint with only the newly completed lane databases. Terminal merge consumes
the checkpoint first. If a manual/source-run resume has complete lane uploads but no
checkpoint, recovery reads the manifest's exact `chain_state.artifact_run_ids` plus
the current run, inventories every artifact, and downloads only exact lane/database
and metadata names. Duplicate names, ambiguous databases, invalid run IDs, and a
database without its same-run metadata fail closed; recovery artifacts and wildcard
prefix matches are never accepted. Every accepted current or historical lane pair
must match the manifest's chain, source, run, artifact name, lane, and coverage
provenance.

Checkpoint roll-forward is copy-plus-delta and never edits the prior generation.
The output path must differ from the previous checkpoint; the builder copies the
base database to a temporary file, merges current complete-lane databases, and
promotes the new file. With no new lane delta, it still emits a physically distinct
copy. In `checkpoint-report.json`, verify `base_database_copied`,
`merged_delta_database_count`, and `copy_fast_path` under `merge_summary` alongside
the generation and coverage fingerprint. Also verify `database_sha256` and
`included_lane_coverage_hashes`; roll-forward rejects a prior database when either
identity no longer matches. `included_lane_workload_contracts` records the exact
player/team/season scope for every included cross-product lane. Its identity excludes
only the cumulative generation digest, so append-only workload growth outside a lane
can roll forward while pair/count/base-identity drift inside the lane still fails.
The accepted lane contract is then rebound to the current cumulative generation.
Metadata schema v3 applies the same binding to each
current lane: lane fields, coverage hash, physical database digest, every applicable
manifest season/type, and endpoint presence must all match before merge. For
player/team/season lanes, the checkpoint independently reloads the content-addressed
discovery workload, verifies every requested pair including zero-pair sentinels, and
requires each expected identity/context in the journal while rejecting stale identities
outside the active workload. Duplicate lane IDs fail before checkpoint indexing.
`contract_blocked` is accepted only when the reported support evidence exactly matches
recomputed rules. Its canonical rows and digest must also match the independent current or
previous manifest chain-state commitment. Cancellation/source resume carries new rows in a
digest-bound pending commitment until the next checkpoint merges and clears them. A prior
checkpoint must match its pointer's chain, source, run,
artifact name, generation, and coverage fingerprint, and its inventory may not contain
any lane outside the current manifest. A failed build remains an attempt-scoped
diagnostic artifact; only a successfully validated database and report are promoted
under the canonical checkpoint name. Valid contract-blocked lanes are recorded in a
separate artifact-bound evidence inventory rather than included in the effective
checkpoint coverage used for terminal assured identity. Staging merge
uses `EXCEPT ALL` overlap subtraction, preserving legitimate duplicate multiplicity
without replaying rows already represented by a prior checkpoint or another delta.

Incomplete lanes upload a longer-retained durable state artifact only after DuckDB
`CHECKPOINT`, WAL removal, regular-file/journal validation, and an attestation that
binds source SHA, chain, lane, coverage hash, and database SHA-256. Metadata is finalized
only after the upload action returns a positive artifact ID and SHA-256 receipt. The next
iteration restores only that exact run/artifact identity and receipt. Finalized metadata gets
one exact-name upload retry before the lane falls back to diagnostics. Player/team/season snapshots use
state-attestation schema v3 to bind the active content-addressed workload generation,
requested pairs,
zero-pair sentinels, and base-identity digest. Extraction-lane cache-prefix and
parent-lane fallbacks are not accepted; split children clear parent pointers. Canonical
metadata schema v3 uploads even when snapshot attestation fails, preserving restore
errors, progress, failure class, root error type, and failed VPN servers for lane
control. Only a pointer with both `state_artifact.attested=true` and
`state_artifact.uploaded=true` plus valid receipt fields is reusable. The database itself
remains diagnostics-only when attestation or upload finalization fails. Active partial pointers
must use
`extraction-lane-recovery-<chain>-<lane>-run-<run>-attempt-<attempt>` with the embedded
positive run ID matching `state_artifact_run_id`; a canonical complete-lane name is not a
partial-resume pointer. An invalid or missing new receipt never replaces an older canonical
recovery pointer or its completed-call/persisted-row baselines. Without prior durable state,
the pointer and baselines remain clear before retry.
The extract job records a 350-minute internal deadline before checkout inside the
360-minute Actions cap and shortens the effective lane timeout as needed to preserve
at least a 20-minute finalization reserve. That
reserve covers status calculation, DuckDB checkpoint and attestation, canonical
metadata, state/database upload and exact-name retry, diagnostics, and VPN disconnect;
the nominal lane timeout is not permission to consume that headroom.

`video_details` and `video_details_asset` classify 1946-47 through 2003-04 as
`contract_blocked` and begin at 2004-05 in full extraction. Run 29195221754 discovery
artifact 8260784820 covered all 290 season/type combinations in the blocked range;
its 89,722 LeagueGameLog rows represented 44,861 games, and every `VIDEO_AVAILABLE`
value was zero. The first three `video_details_asset` lanes in the same run completed
8,902 calls and persisted zero rows. This evidence-bounded rule removes only the
unavailable range; both endpoints recursively parse every returned result set and
retain endpoint, result-set, player/team/season/type, and context-measure provenance.
The extraction contract unions all 78 measures exposed across the exact `nba_api`
1.11.4 runtime and upstream docs/tools from 2004-05 onward, groups at most three per
lane, and classifies pre-2019 PlayIn,
pre-1950-51 All Star, and cancelled 1998-99 All Star scopes as
`upstream_unavailable`. A non-2xx response is rejected before JSON parsing. HTTP
429/5xx responses and equivalent JSON error envelopes are transient; malformed
success payloads are contract failures and are not staged. The asset runner
additionally limits the failure blast radius: ten calls per durable chunk, two
concurrent requests, a 15-second request timeout, no per-call retry loop, abort after
a fully failed newly attempted chunk, and a 600-second no-completed-chunk watchdog. A
successful empty chunk is persisted and then journaled before the next chunk starts.

Revalidate either cutoff boundary with the support rule's exact one-season probe,
changing only the endpoint name:

```bash
uv run nbadb backfill run --extract-only --verbose \
  --pattern player_team_season \
  --endpoint video_details \
  --seasons 2003:2004 \
  --summary-path artifacts/extraction/extract-summary.json
```

`matrix_batch_size` requests how many lane jobs enter one workflow iteration and
defaults to GitHub's 256-job matrix ceiling. Direct mode may use that full value;
both `network_mode=vpn` and `network_mode=auto` planning apply an additional cap of
`vpn_parallelism * 32`, so the default configured value of two plans at most 64 jobs per
iteration. The later gate can admit fewer configured tunnels, and token-derived
authentication executes one simultaneous matrix job without shrinking the already
planned batch. `matrix_batch_size` remains independent from
`vpn_parallelism` and `direct_parallelism`, which control simultaneous runners and
therefore API pressure. VPN mode defaults to two simultaneous lanes. Each active
lane runs on a separate GitHub-hosted runner and tunnel. Probe-verified
preflight/discovery hosts are reused only by their assigned parallel slot; unfilled
slots retain disjoint recommendation partitions. Those preferences do not attest
unique VPN exit IPs across runners. When alternatives exist, the scheduler prevents
an endpoint-homogeneous six-lane runner window, which distributes endpoint pressure
without changing that VPN assurance boundary. All lanes still consume the same
account's connection allowance. Configured credentials must prove the requested
simultaneous tunnel count in a small concurrent admission gate before the matrix
starts; a failed gate skips fan-out and uploads per-probe diagnostics.
The subsequent quarantine job must also pass before `extract`; inspect its
run-attempt-scoped report when marker provenance or failure aggregation is rejected.

If a later extraction connector returns exact `vpn_auth_failure`, that lane publishes
`full-extraction-vpn-auth-circuit-run-<run_id>-attempt-<attempt>`. Every queued lane
checks for that immutable marker before authenticating. Because the check runs inside
each admitted matrix job, siblings already past it can each receive one rejection before
the marker is visible. The check validates the exact artifact identity, workflow
run/source, archive SHA-256, safe single-file ZIP, and canonical marker payload. Only a
verified marker produces retry-neutral `vpn_auth_circuit_open` metadata. Lookup or
validation failure still prevents a connector call, but produces
`vpn_auth_circuit_check_failed`, consumes one bounded `runner_infrastructure` retry, and
does not open the provider circuit or suppress a healthy redispatch. Any rejecting lane
or valid circuit-open sibling opens the lane-control circuit, so the workflow checkpoints
completed lanes and does not self-dispatch the next iteration.
`vpn_connect_timeout` and `vpn_network_error` do not open this circuit. A failed marker
lookup is also fail-closed: the lane does not authenticate while GitHub coordination is
unavailable. Only
one VPN/auto full-extraction workflow can run at a time; direct-mode chains retain
chain/iteration-scoped concurrency. Token-derived mode limits execution to one
simultaneous matrix job; it does not reduce the planned batch to one lane.
[NordVPN documents ten simultaneous devices and requires different protocols when
multiple devices use the same server](https://support.nordvpn.com/hc/en-us/articles/19476515228305-How-many-devices-can-I-use-with-NordVPN),
so raise `vpn_parallelism` only when the account has enough unused slots and the
selected server/protocol combinations remain distinct. If the capacity gate fails,
inspect `vpn-capacity-diagnostics-<chain_id>-<lane_index>` and rerun the complete workflow
attempt after repairing the account or network issue:

```bash
set -euo pipefail

: "${REPOSITORY:?export REPOSITORY=owner/repository}"
: "${RUN_ID:?export RUN_ID to the positive GitHub Actions run ID}"

[[ "$REPOSITORY" =~ ^[^/[:space:]]+/[^/[:space:]]+$ ]]
[[ "$RUN_ID" =~ ^[1-9][0-9]*$ ]]

gh run rerun "$RUN_ID" --repo "$REPOSITORY"
```

Do not use `gh run rerun --failed`: successful peers would not republish markers for the
new run attempt, so the current-attempt barrier could never become complete. No
checkpoint or next-manifest
handoff exists because extraction was never admitted. If the later auth circuit opens,
retain the checkpoint and next-manifest artifacts, wait for the provider cooldown, and
resume the stopped chain with the exact handoff below. Increase parallelism again only
after a sustained clean wave, not from the nominal device allowance alone.

```bash
set -euo pipefail

: "${REPOSITORY:?export REPOSITORY=owner/repository}"
: "${SOURCE_RUN_ID:?export SOURCE_RUN_ID to the stopped run ID}"
: "${CHAIN_ID:?export CHAIN_ID to the exact extraction chain ID}"
: "${WORKFLOW_REF:?export WORKFLOW_REF to the trusted branch}"
: "${WORKFLOW_SHA:?export WORKFLOW_SHA to the pinned 40-character commit SHA}"

command -v gh >/dev/null
command -v jq >/dev/null
[[ "$REPOSITORY" =~ ^[^/[:space:]]+/[^/[:space:]]+$ ]]
[[ "$SOURCE_RUN_ID" =~ ^[1-9][0-9]*$ ]]
[[ "$WORKFLOW_SHA" =~ ^[0-9a-fA-F]{40}$ ]]

workflow_sha="$(printf '%s' "$WORKFLOW_SHA" | tr '[:upper:]' '[:lower:]')"
work_dir="$(mktemp -d)"
trap 'rm -rf -- "$work_dir"' EXIT
inventory="$work_dir/artifacts.json"
match="$work_dir/next-manifest-artifact.json"
resume_dir="$work_dir/resume"

gh api --method GET \
  -H "Accept: application/vnd.github+json" \
  "/repos/${REPOSITORY}/actions/runs/${SOURCE_RUN_ID}/artifacts" \
  -f per_page=100 \
  --paginate --slurp > "$inventory"
jq -e --arg prefix "full-extraction-next-manifest-${CHAIN_ID}-iter-" '
  [.[].artifacts[]
    | select(.expired == false)
    | select((.name // "") | startswith($prefix))] as $matches
  | if ($matches | length) != 1 then
      error("expected exactly one unexpired next-manifest artifact")
    elif (($matches[0].name | ltrimstr($prefix)) | test("^[1-9][0-9]*$"))
      | not then
      error("next-manifest artifact has an invalid iteration suffix")
    else
      $matches[0]
    end
' "$inventory" > "$match"

  jq -er --arg prefix "full-extraction-next-manifest-${CHAIN_ID}-iter-" \

      ((.key | type) == "string")

  and ($manifest.iteration_budget | positive_integer)
```

Derive the candidate `next_iteration` from the one exact artifact name; do not guess
it. Validate chain, source, full-manifest coverage, and current checkpoint ancestry in
`next-manifest.json`, then cross-bind the report's artifact, next iteration, active
count, budget, checkpoint generation/name, and checkpoint coverage hash to that
manifest. The auth-circuit counts and reasons must agree, and the serialized redispatch
payload must remain inside the source-enforced 60,000-byte bound. Artifact discovery
happens first because both authoritative files are inside that artifact, and requiring
one unexpired prefix match avoids a circular handoff.

With `max_iterations=auto`, the first manifest derives one numeric `iteration_budget`
from total remaining matrix dispatch credits, per-lane retry depth, and bounded split
descendants. Child manifests preserve that value and never extend it. Redispatch still
passes the literal `auto` input while the fixed manifest cap remains authoritative.
Each lane also carries a cumulative `zero_progress_streak`. It resets only when completed-call
or persisted-row counters advance and a valid, durably uploaded state-artifact receipt backs
that progress. Unchanged counters increment the streak. Reported growth without a valid receipt
is diagnostics-only and also increments the streak, even if the failure class alternates between
attempts. This keeps mixed transport, response-contract, and infrastructure failures inside one
bounded no-progress budget.
Before `gh workflow run`, the dispatch job queries all paginated
workflow-dispatch runs and refuses an active or successful display-title match for
`Full Extraction chain=<chain_id> iteration=<next>`. The job-level concurrency key
serializes dispatch attempts for the current chain iteration as an additional guard.
VPN/auto workflow concurrency is a repository-wide FIFO queue so separate chains
cannot consume the same account slots concurrently or displace an already-pending
child; direct workflows remain scoped to chain plus iteration. Active or successful
exact-title children block redispatch, but
failed, cancelled, or timed-out historical runs are recorded and excluded when acknowledging the
one newly created child run ID and URL. The first exact-title match is provisional:
the dispatcher performs a stabilization poll and acknowledges the child only when the
new child set remains exactly one. Normal data, metadata, diagnostics, and checkpoint
upload-artifact steps use overwrite semantics so rerun jobs can replace their prior
logical artifacts. Each lane's initial capacity-marker upload is immutable and uses
`overwrite: false`; only after that lane-unique first upload reports failure may the
same lane make one exact-name retry with `overwrite: true`. Auth-circuit markers remain
shared first-writer artifacts: their retry also uses `overwrite: false` and can succeed
only by resolving the exact sibling marker.
If a parent is interrupted after dispatch but before child acknowledgement, its exit
trap attempts to cancel every newly created exact-title child. Once the child is
acknowledged it is independently durable and must be monitored by run ID. Manual
cancellation also prevents queued network jobs from starting and lets the lane wrapper
make a bounded signal-driven cleanup attempt, but GitHub may remove an ephemeral runner
before subsequent metadata or artifact steps execute. Recover only from the latest
attested upload or checkpoint; cancellation itself is not durability evidence.

The pinned `workflow_sha` must be a full commit on the trusted `workflow_ref` branch;
publish runs also prove default-branch ancestry. Every uploaded lane manifest carries
that exact source SHA and its chain identity; manual handoff, chained handoff, and
source-run resume all fail before network work when either value differs.
Publish-capable runs perform an
early credential/exact-version preflight before lane fan-out and repeat it after
terminal assurance. Terminal assurance has read-only contents permission and emits
an exact run/attempt-scoped artifact. Only the separate publisher job receives write
permission and Kaggle secrets, and all full/daily/monthly publishers enter the shared
FIFO `queue: max` group. The terminal export also creates
`assured-artifact-manifest.json`, but only after every export completes and
`nbadb scan --fail-on error --full-publication --checkpoint-report <path>
--checkpoint-manifest <path> --checkpoint-dir <dir> --checkpoint-chain-id <id>
--checkpoint-source-sha <sha>` invokes the canonical checkpoint verifier and validates
both general quality and the terminal checkpoint's complete lane/run/coverage inventory,
including canonical blocked evidence, while requiring every declared nonempty silver/gold
domain anchor and row-preserving cardinality pair. Its sorted file hashes bind the data tree to the
source commit, chain ID, and validated effective checkpoint coverage; separately
artifact-bound blocked evidence cannot substitute for extracted coverage. The
publisher resolves the exact GitHub artifact ID, verifies the downloaded archive's
SHA-256 against the upload result, and then recomputes the assured manifest before
requiring `nbadb upload --full-publication`, which independently validates the declared
terminal assurance report and its chain, source, effective coverage, checkpoint, and
blocked-evidence identities, exact resource kinds, and DuckDB/SQLite/CSV/Parquet
table-schema-row parity before publishing both assurance files as Kaggle resources
and repeating their provenance in publication marker schema v2. Generic
`--verify-remote` remains available for exact-version readback of daily or monthly
updates without asserting full-extraction terminal provenance. Generated metadata covers
all 254 runtime transform outputs. The default
branch must equal the pinned source immediately before upload. A failed-job rerun may
also accept its single exact metadata-only child, but only when the regenerated file
is byte-identical. The idempotent metadata push is the publisher's final step, after
exact remote verification and receipt artifacts succeed. If a source run already has
zero active lanes, resume with its original `chain_id` and `resume_source_run_id`.
The workflow replays an attested terminal checkpoint when present; if cancellation
landed before checkpoint promotion, it rebuilds from the source run's exact attested
complete-lane artifacts instead of rerunning those lanes.

Use `publish=false` when a full-coverage Actions canary is worth the extra run. It
exercises terminal transform, export, and full-publication hard scan without permitting
metadata or Kaggle publication. For a narrow custom manifest, set `targeted_smoke=true` with
`network_mode=vpn`, `publish=false`, `max_iterations=1`, and
`retry_pipeline_failures=false`. The workflow then skips global merge/scan and
redispatch, and its dedicated assurance job passes only when the manual manifest has
exactly one lane, lane control reports one complete `resume_only` lane with no active
or contract-blocked work, and the checkpoint is terminal-ready. This proves the
extractor/VPN/checkpoint path only; it is not full-dataset assurance. Use direct mode
only when a separate connectivity probe succeeds. Direct mode skips NordVPN
setup, uses `direct_parallelism` instead of VPN fan-out, caps high-volume
temporal lanes (`date`, `game`, `player_season`, `team_season`, and
`player_team_season`) before writing timeout metadata, writes `direct-no-vpn`
into lane metadata, and does not require VPN diagnostic artifacts for failed
lanes. Start direct fan-out low (`1` or `2`) for smoke validation, then raise it
toward the workflow ceiling only after lane metadata proves the direct path is
viable; use `network_mode=auto` only after future iterations should prefer VPN
again when preflight recovers.

### 4) Remediate

- Retry `needs_resume` lanes by default. Keep a parent lane intact when the latest
  attempt added calls or rows and its attested snapshot was durably uploaded with
  a finalized artifact receipt; this preserves journal skips for transport-class
  partials and progress-making timeouts. A timeout-class lane is split only after
  an attempt adds no durable progress. Short `game`/`date` spans split to
  one-season children, while `season`, `player_season`, `team_season`, and
  `player_team_season` lanes split by their lane policy span.
- Let the workflow retry a failed complete or partial lane-state upload once with
  the exact same artifact name. Only the successful attempt's positive artifact ID
  and SHA-256 receipt may finalize resumable state; two failed attempts downgrade
  the snapshot to diagnostics-only runner-infrastructure evidence.
- Do not retry `contract_blocked` lanes. Add or update support rules only when endpoint analysis or local probes supply evidence and a revalidation command.
- Treat `pipeline_failure` by failure class. Transport, VPN, runner-infrastructure, response-contract, timeout-progress, and timeout-stalled lanes have bounded retry budgets; the cumulative zero-progress budget continues across alternating classes. Application failures should normally be fixed before relaunch. Set `retry_pipeline_failures=false` when you want an audit-only run that stops instead of retrying classed failures.
- If discovery seeding fails, download the run/attempt-scoped discovery recovery artifact and inspect `artifacts/discovery/discovery-seed-summary.json`. Repair only the exact `missing_units` player seasons/scopes, game combinations, or player-team-season pairs. Keep the recovery name distinct; resume it only through `lane_manifest_run_id`, which reruns integrity, exact-coverage, and canonical bundle gates. Both transport and response-contract/validation failures should consume the bounded recovery attempts; a first response-shape failure must not terminate the remaining budget. Do not treat a broader aggregate cache as proof of an exact missing scope.
- If `nba-probe-status` is `timeout`, `http_error`, `request_failed`, `invalid_content`, `stack_timeout`, `stack_transport_failed`, or `stack_budget_exhausted`, treat that server as NBA-blocked even when its exit-IP probe passed. Confirm the server appears in `failed-servers-json`; do not start discovery on that tunnel.
- If `nba-probe-status` is `stack_unavailable`, `stack_runtime_error`, `stack_contract_error`, or `stack_invalid_attestation`, stop and fix the checked-out runtime or contract. These are host-independent fatal failures and must not rotate through or quarantine healthy VPN servers.
- If checkpoint promotion fails, fix or regenerate the checkpoint before terminal merge. Do not rename attempt-scoped diagnostics to the canonical checkpoint. Use lane-artifact fallback only as an explicit recovery decision after confirming exact chain/source/run/artifact/generation/coverage provenance and proving that neither the prior checkpoint nor its blocked-evidence inventory names a lane outside the current manifest. If both checkpoint and lane artifacts are missing, relaunch from the latest `full-extraction-next-manifest-*` artifact rather than starting from scratch.
- If redispatch reports an existing exact chain/iteration run, inspect that run instead of enqueueing another copy. Do not treat the first visible child as acknowledged until the stabilization poll confirms exactly one new exact-title run. If the numeric iteration budget is exhausted while work remains, resume manually from the uploaded next-manifest and checkpoint artifacts after deciding whether to raise the cap.

## Kaggle and common CLI/CI misses

| Command or area                       | Symptom                                      | Fix                                                                                                                                                                                |
| ------------------------------------- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `uv sync`                             | Resolver / lock mismatch                     | Regenerate the lock, rerun sync, and commit the lockfile CI expects                                                                                                                |
| `uv run nbadb download` / `upload`    | Download or upload failure                   | Ensure Kaggle credentials are available to the environment and the dataset slug is correct                                                                                         |
| `uv run nbadb upload --verify-remote` | Remote publish stays pending                 | Inspect exact `baseline.version`, `resolved_version`, `observations`, and status in `logs/kaggle/kaggle-upload-manifest.json`; do not re-upload while reconciliation is unresolved |
| `nbadb scan --report-path ...`        | `database not found`                         | Initialize or download the dataset before running quality checks                                                                                                                   |
| `actions/upload-artifact`             | “No files were found with the provided path” | Verify the upstream command wrote exactly the artifact path the workflow expects                                                                                                   |
| docs install/build step               | lockfile or dependency drift                 | Refresh docs dependencies and commit the matching lockfile changes                                                                                                                 |

For remote verification, the positive integer returned by Kaggle's marker resolver
is the version evidence; a cache path such as `versions/999` is not. After resolution,
the client paginates the exact version's complete API inventory and downloads one
listed file at a time for SHA-256 comparison, removing it before the next file. Baseline
handling is intentionally narrow:

| Manifest evidence                                                                              | Meaning                                                                                          | Safe action                                                               |
| ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------- |
| `baseline.state=marker_present` and `matches_expected=true`                                    | Kaggle already exposes this publish key and bundle at `baseline.version`.                        | The command records `reconciled_existing_remote` and skips upload.        |
| `baseline.state=bootstrap_marker_missing`, `marker_status_code=404`, and `upload_allowed=true` | The marker alone is absent; the dataset metadata API supplied an exact baseline version.         | Allow the single bootstrap upload, then wait for marker readback.         |
| `status=baseline_reconciliation_failed`                                                        | Baseline lookup timed out, returned a non-404 error, or could not supply exact version evidence. | Fix the lookup/authentication problem; `upload_attempts` remains `0`.     |
| `status=publication_reconciliation_required`                                                   | One upload was attempted, but the expected marker was not visible before the deadline.           | Wait and rerun verification; do not assume the remote version was absent. |
| `status=bootstrap_reconciliation_required`                                                     | A prior marker-404 upload for this dataset is still unresolved.                                  | Reconcile the prior marker identity; the client refuses every new bundle. |

A later run can safely reconcile a now-visible matching marker. If it matches an
older unresolved bundle, the client resolves that record before considering the
current bundle; if it matches the current bundle, upload is skipped. If the marker
is still 404 or nonmatching, the client fails before another upload even when the
current version differs from the original baseline.
For any publish retry, confirm that the job restored the shared
`nbadb-kaggle-publication-state-*` cache and that its publication artifact contains
`logs/kaggle/kaggle-publication-state.json`; do not retry publication if that
durable evidence is unexpectedly absent after an ambiguous upload. Publishing
workflows reject non-default refs because Actions cache visibility is ref-scoped.

For end-to-end acceptance, enumerate every exact-title chain run across all API pages
instead of monitoring only the first run. Recovery can leave several run IDs for one
iteration, so group that history by iteration and choose the highest run ID as the
authoritative attempt. Every superseded attempt must be completed with exactly
`failure`, `cancelled`, or `timed_out`; an older success, active attempt, empty
conclusion, or any other conclusion invalidates the chain. Authoritative iterations
must remain unique and contiguous from the expected first iteration.

Require the common guard, planning, and publication-preflight jobs in every
authoritative attempt. A normal nonterminal run has overall `success`, all extract jobs
successful, and successful lane control, checkpoint, and dispatch. A bounded
infrastructure redispatch has overall `failure`, only completed success/failure extract
jobs with at least one failure, successful lane control, checkpoint, and dispatch, a
closed auth circuit, and positive `runner_infrastructure` evidence in the exact bounded
chain-dispatch report. A validated auth-circuit handoff has the same constrained extract
failure and successful lane control/checkpoint, but its circuit is open and dispatch
skips for manual recovery. Both nonterminal failure branches skip merge and publish.
The terminal authoritative attempt must have overall `success`, skip dispatch, and pass
merge terminal assurance plus publish. Every nonterminal branch cross-validates the
exact next-manifest artifact against its dispatch report for next iteration, active
count, iteration budget, source/chain, checkpoint generation/artifact, and checkpoint
coverage identity. Download
`full-extraction-kaggle-publication-receipt-<chain_id>` and accept deployment only when
`logs/kaggle/kaggle-upload-manifest.json` records a successful or reconciled exact
publication, a positive resolved Kaggle version, and completed exact remote inventory
and SHA-256 readback. Its preflight provenance, normalized terminal assurance, and
remote publication marker must each independently match the expected chain, source,
and terminal coverage fingerprint. A green upload submission without that receipt is
not full publication assurance.

```bash
    def positive_integer:
      type == "number" and . > 0 and . == floor;

    [.[].artifacts[]
      | select(.expired == false and (.name // "") == $name)]
    | if length != 1 then
        error("expected one exact nonterminal next-manifest artifact")
      elif ((.[0].id | positive_integer) | not)
        or ((.[0].size_in_bytes | positive_integer) | not)
        or ((.[0].archive_download_url // "") == "") then
        error("next-manifest artifact identity is incomplete")
      else
        .[0]
      end
  ' "$artifact_pages" > "$artifact_match"

  mkdir -p "$artifact_dir"
  gh run download "$run_id" \
    --repo "$REPOSITORY" \
    --name "$artifact_name" \
    --dir "$artifact_dir"

  local manifest_count
  local report_count
  local next_manifest
  local dispatch_report
  manifest_count="$(
    find "$artifact_dir" -type f -name next-manifest.json -print \
      | wc -l \
      | tr -d '[:space:]'
  )"
  report_count="$(
    find "$artifact_dir" -type f -name chain-dispatch.json -print \
      | wc -l \
      | tr -d '[:space:]'
  )"
  [ "$manifest_count" -eq 1 ]
  [ "$report_count" -eq 1 ]
  next_manifest="$(find "$artifact_dir" -type f -name next-manifest.json -print)"
  dispatch_report="$(find "$artifact_dir" -type f -name chain-dispatch.json -print)"

  jq -e \
    --arg mode "$mode" \
    --arg chain "$CHAIN_ID" \
    --arg sha "$workflow_sha" \
    --arg run_id "$run_id" \
    --arg artifact "$artifact_name" \
    --argjson iteration "$next_iteration" \
    --slurpfile dispatch "$dispatch_report" '
    def positive_integer:
      type == "number" and . > 0 and . == floor;
    def nonnegative_integer:
      type == "number" and . >= 0 and . == floor;
    def sha256:
      type == "string" and test("^[0-9a-f]{64}$");
    def count_map:
      type == "object"
      and all(to_entries[];
        ((.key | type) == "string")
        and ((.key | length) > 0)
        and (.value | positive_integer));

    . as $manifest
    | (if ($dispatch | length) == 1 then
        $dispatch[0]
      else
        error("expected exactly one chain-dispatch report")
      end) as $report
    | ($manifest.chain_state // null) as $state
    | ($report.vpn_auth_circuit_deferred_lane_count // -1) as $deferred
    | ($report.vpn_auth_circuit_check_failed_lane_count // -1) as $check_failed
    | ($report.vpn_auth_circuit_rejection_lane_count // -1) as $rejections
    | ($report.failure_class_counts.runner_infrastructure // 0) as $infrastructure
    | type == "object"
    and ($state | type) == "object"
    and ($report | type) == "object"
    and $manifest.chain_id == $chain
    and $manifest.workflow_source_sha == $sha
    and ($manifest.coverage_fingerprint | sha256)
    and ($manifest.active_lane_count | positive_integer)
    and ($manifest.iteration_budget | positive_integer)
    and $state.latest_checkpoint_run_id == $run_id
    and ($state.latest_checkpoint_generation | positive_integer)
    and ($state.latest_checkpoint_coverage_hash | sha256)
    and ($state.latest_checkpoint_artifact_name
      == ("full-extraction-checkpoint-" + $chain + "-iter-"
        + ($state.latest_checkpoint_generation | tostring)))
    and $report.artifact_name == $artifact
    and $report.next_iteration == $iteration
    and $report.active_lane_count == $manifest.active_lane_count
    and $report.iteration_budget == $manifest.iteration_budget
    and $report.iteration_budget >= $report.next_iteration
    and $report.checkpoint_artifact_name == $state.latest_checkpoint_artifact_name
    and $report.checkpoint_generation == $state.latest_checkpoint_generation
    and $report.checkpoint_coverage_hash == $state.latest_checkpoint_coverage_hash
    and ($report.redispatch_payload_bytes | positive_integer)
    and $report.redispatch_payload_bytes <= 60000
    and ($report.resume_only_lane_count | nonnegative_integer)
    and ($report.contract_blocked_lane_count | nonnegative_integer)
    and ($report.split_lane_count | nonnegative_integer)
    and ($deferred | nonnegative_integer)
    and ($check_failed | nonnegative_integer)
    and ($rejections | nonnegative_integer)
    and ($report.outcome_counts | count_map)
    and ($report.failure_reason_counts | count_map)
    and ($report.failure_class_counts | count_map)
    and (($report.failure_reason_counts.vpn_auth_circuit_check_failed // 0)
      == $check_failed)
    and $infrastructure >= $check_failed
    and (if $mode == "normal" then
      $report.vpn_auth_circuit_open == "false"
      and $deferred == 0
      and $rejections == 0
    elif $mode == "infrastructure" then
      $report.vpn_auth_circuit_open == "false"
      and $deferred == 0
      and $rejections == 0
      and ($infrastructure | positive_integer)
      and (([$report.failure_class_counts[]] | add // 0)
        == ([$report.failure_reason_counts[]] | add // 0))
    elif $mode == "auth_handoff" then
      $report.vpn_auth_circuit_open == "true"
      and (($deferred + $rejections) > 0)
      and (($report.failure_reason_counts.vpn_auth_circuit_open // 0)
        == $deferred)
      and (($report.failure_reason_counts.vpn_auth_failure // 0) == $rejections)
    else
      error("unsupported nonterminal validation mode")
    end)
  ' "$next_manifest" >/dev/null
}

while IFS=$'\t' read -r run_id iteration head_sha conclusion run_url superseded_count; do
  jobs_pages="$work_dir/jobs-${run_id}.json"
  gh api --method GET \
    -H "Accept: application/vnd.github+json" \
    "/repos/${REPOSITORY}/actions/runs/${run_id}/jobs" \
    -f filter=latest \
    -f per_page=100 \
    --paginate --slurp > "$jobs_pages"

  terminal=false
  if [ "$run_id" = "$terminal_run_id" ]; then
    terminal=true
  fi

  dispatch_conclusion="$(
    jq -er '
      [.[].jobs[] | select((.name // "") == "dispatch_next")]
      | if length == 1 then
          .[0].conclusion
        else
          error("expected exactly one dispatch_next job")
        end
    ' "$jobs_pages"
  )"

  branch=invalid
  if [ "$terminal" = true ]; then
    branch=terminal
  elif [ "$conclusion" = success ] && [ "$dispatch_conclusion" = success ]; then
    branch=normal
  elif [ "$conclusion" = failure ] && [ "$dispatch_conclusion" = success ]; then
    branch=infrastructure
  elif [ "$conclusion" = failure ] && [ "$dispatch_conclusion" = skipped ]; then
    branch=auth_handoff
  fi

  if [ "$terminal" = false ] && [ "$branch" != invalid ]; then
    validate_nonterminal_dispatch "$run_id" "$iteration" "$branch"
  fi

  jq -e \
    --arg run_id "$run_id" \
    --arg run_conclusion "$conclusion" \
    --arg branch "$branch" \
    --argjson terminal "$terminal" \
    '
    [.[].jobs[]] as $jobs
    | def matches($base):
        [$jobs[]
          | select((.name // "") == $base
            or ((.name // "") | startswith($base + " (")))];
      def require_success($base):
        matches($base) as $matches
        | if ($matches | length) == 0 then
            error("run " + $run_id + " is missing required job " + $base)
          elif all($matches[];
            .status == "completed" and .conclusion == "success") then
            true
          else
            error("run " + $run_id + " has an unsuccessful required job " + $base)
          end;
      def require_skipped($base):
        matches($base) as $matches
        | if ($matches | length) == 0 then
            error("run " + $run_id + " is missing expected skipped job " + $base)
          elif all($matches[];
            .status == "completed" and .conclusion == "skipped") then
            true
          else
            error("run " + $run_id + " did not skip job " + $base)
          end;
      def require_capacity_result:
        matches("vpn_capacity") as $matches
        | if ($matches | length) == 0 then
            error("run " + $run_id + " is missing vpn_capacity")
          elif all($matches[];
            .status == "completed" and .conclusion == "success") then
            true
          elif all($matches[];
            .status == "completed" and .conclusion == "skipped") then
            true
          else
            error("run " + $run_id + " has an invalid vpn_capacity result")
          end;
      def require_run_conclusion($expected):
        if $run_conclusion == $expected then
          true
        else
          error("run " + $run_id + " must conclude " + $expected)
        end;
      def require_branch($expected):
        if $branch == $expected then
          true
        else
          error("run " + $run_id + " did not match branch " + $expected)
        end;
      def require_active_dependencies:
        [
          require_success("preflight"),
          require_success("discovery_seed"),
          require_capacity_result,
          require_success("vpn_quarantine")
        ] | all;
      def require_normal_active_iteration:
        [
          require_active_dependencies,
          require_success("extract"),
          require_success("lane_control"),
          require_success("checkpoint")
        ] | all;
      def require_failed_extracts($label):
        matches("extract") as $matches
        | if ($matches | length) == 0 then
            error("run " + $run_id + " is missing extract jobs")
          elif (all($matches[];
            .status == "completed"
            and (.conclusion == "success" or .conclusion == "failure"))
            and any($matches[]; .conclusion == "failure")) then
            true
          else
            error("run " + $run_id
              + " " + $label
              + " requires completed success/failure extract jobs"
              + " with at least one failure")
          end;
      def require_failed_active_iteration($label):
        [
          require_active_dependencies,
          require_failed_extracts($label),
          require_success("lane_control"),
          require_success("checkpoint")
        ] | all;
      def require_terminal_source:
        matches("terminal_replay") as $matches
        | if ($matches | length) == 0 then
            error("run " + $run_id + " is missing terminal_replay")
          elif all($matches[];
            .status == "completed" and .conclusion == "success") then
            true
          elif all($matches[];
            .status == "completed" and .conclusion == "skipped") then
            require_normal_active_iteration
          else
            error("run " + $run_id + " has an invalid terminal_replay result")
          end;

      [
        require_success("workflow_guard"),
        require_success("plan"),
        require_success("publication_preflight"),
        if $terminal then
          require_branch("terminal"),
          require_run_conclusion("success"),
          require_success("merge"),
          require_success("publish"),
          require_skipped("dispatch_next"),
          require_terminal_source
        else
          if $branch == "auth_handoff" then
            require_branch("auth_handoff"),
            require_run_conclusion("failure"),
            require_failed_active_iteration("auth handoff"),
            require_skipped("dispatch_next")
          elif $branch == "infrastructure" then
            require_branch("infrastructure"),
            require_run_conclusion("failure"),
            require_failed_active_iteration("infrastructure redispatch"),
            require_success("dispatch_next")
          elif $branch == "normal" then
            require_branch("normal"),
            require_run_conclusion("success"),
            require_normal_active_iteration,
            require_success("dispatch_next")
          else
            error("run " + $run_id + " has no accepted nonterminal branch")
          end,
          require_skipped("merge"),
          require_skipped("publish")
        end
      ] | all
  ' "$jobs_pages" >/dev/null

  printf 'validated iteration=%s run_id=%s branch=%s superseded=%s sha=%s conclusion=%s url=%s\n' \
    "$iteration" "$run_id" "$branch" "$superseded_count" \
    "$head_sha" "$conclusion" "$run_url"
done < <(
  jq -r '.[]
    | [.id, .iteration, .head_sha, .conclusion, .html_url,
      (.superseded_attempts | length)]
    | @tsv' \
    "$chain_runs"
)

receipt="$work_dir/receipt"
mkdir -p "$receipt"
gh run download "$terminal_run_id" \
  --repo "$REPOSITORY" \
  --name "full-extraction-kaggle-publication-receipt-${CHAIN_ID}" \
  --dir "$receipt"

manifest_count="$(
  find "$receipt" -type f -name kaggle-upload-manifest.json -print \
    | wc -l \
    | tr -d '[:space:]'
)"
[ "$manifest_count" -eq 1 ]
manifest="$(find "$receipt" -type f -name kaggle-upload-manifest.json -print)"

jq -e \
  --arg chain "$CHAIN_ID" \
  --arg sha "$workflow_sha" \
  --arg coverage "$coverage_fingerprint" '
  def positive_integer:
    type == "number" and . > 0 and . == floor;
  def nonnegative_integer:
    type == "number" and . >= 0 and . == floor;
  def sha256:
    type == "string" and test("^[0-9a-f]{64}$");

  .preflight.provenance as $preflight_provenance
  | .preflight.terminal_assurance as $terminal_assurance
  | .remote_readback.provenance as $remote_provenance
  | .publication.resolved_version as $publication_version
  | .remote_readback.resolved_version as $readback_version
  | .remote_readback.resource_verification.version as $resource_version
  | .remote_readback.resource_verification as $verified
  | .preflight.expected_staged as $expected_tree
  | .post_upload as $post_upload
  |
  ((.status == "uploaded_remote_verified")
    or (.status == "reconciled_existing_remote"))
  and ($preflight_provenance | type) == "object"
  and ($terminal_assurance | type) == "object"
  and ($remote_provenance | type) == "object"
  and .remote_readback.schema_version == 2
  and $terminal_assurance.schema_version == 2
  and $preflight_provenance.chain_id == $chain
  and $terminal_assurance.chain_id == $chain
  and $remote_provenance.chain_id == $chain
  and $preflight_provenance.source_sha == $sha
  and $terminal_assurance.source_sha == $sha
  and $remote_provenance.source_sha == $sha
  and $preflight_provenance.coverage_fingerprint == $coverage
  and $terminal_assurance.coverage_fingerprint == $coverage
  and $remote_provenance.coverage_fingerprint == $coverage
  and ($preflight_provenance.data_tree_fingerprint | sha256)
  and ($remote_provenance.data_tree_fingerprint
    == $preflight_provenance.data_tree_fingerprint)
  and ($publication_version | positive_integer)
  and ($readback_version | positive_integer)
  and ($resource_version | positive_integer)
  and ($publication_version == $readback_version)
  and ($readback_version == $resource_version)
  and (.remote_readback.verification_mode == "publication_marker")
  and (.publication.resource_verification_mode
    == "exact_api_inventory_and_sha256_full_readback")
  and ($verified.content_identity == "sha256_full_readback")
  and (.remote_readback.bundle_fingerprint | sha256)
  and (.remote_readback.bundle_fingerprint == .preflight.fingerprint)
  and (.remote_readback.bundle_fingerprint
    == .publication.expected_bundle_fingerprint)
  and (.remote_readback.resource_count | positive_integer)
  and (.preflight.resource_count | positive_integer)
  and (.remote_readback.resource_count == .preflight.resource_count)
  and (.remote_readback.resource_count
    == (.remote_readback.resources | length))
  and (.remote_readback.resource_bytes | nonnegative_integer)
  and (.preflight.resource_bytes | nonnegative_integer)
  and (.remote_readback.resource_bytes == .preflight.resource_bytes)
  and (.remote_readback.resource_bytes
    == ([.remote_readback.resources[].bytes] | add))
  and ($verified.file_count | positive_integer)
  and ($verified.api_file_count | positive_integer)
  and ($expected_tree.file_count | positive_integer)
  and ($post_upload.file_count | positive_integer)
  and ($verified.file_count == $verified.api_file_count)
  and ($verified.file_count == $expected_tree.file_count)
  and ($verified.file_count == $post_upload.file_count)
  and ($verified.bytes | positive_integer)
  and ($expected_tree.bytes | positive_integer)
  and ($post_upload.bytes | positive_integer)
  and ($verified.bytes == $expected_tree.bytes)
  and ($verified.bytes == $post_upload.bytes)
  and ($verified.fingerprint | sha256)
  and ($verified.fingerprint == $expected_tree.fingerprint)
  and ($verified.fingerprint == $post_upload.fingerprint)
  and ($verified.fingerprint == .publication.expected_fingerprint)
' "$manifest"
```

## Best next move after a fix

1. Rerun the narrowest failing command first.
2. Re-open the generated artifact or report, not just the terminal output.
3. Only rerun a wider pipeline command once the local symptom has actually cleared.
