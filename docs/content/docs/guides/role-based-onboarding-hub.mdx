## Route board in one glance

| Role today                  | Start here                                                | First useful action                                                                                                   | Leave with                                                    |
| --------------------------- | --------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| Contributor                 | [Installation](/docs/installation)                        | Run `uv sync --extra dev` so the local toolchain is real before you touch code or docs                                | A working dev environment and the right follow-up page        |
| Analyst without local setup | [SQL Playground](/docs/playground)                        | Run one sample query in the browser and change a filter, aggregate, or sort                                           | A query shape you can port to live data                       |
| Analyst with live data      | [Analytics Quickstart](/docs/guides/analytics-quickstart) | Connect to `nba.duckdb` and stop once `SHOW TABLES` confirms the floor                                                | One working answer or reusable notebook cell                  |
| Operator                    | [Daily Updates](/docs/guides/daily-updates)               | Run `nbadb status --output-format json` so you know the current state before you choose `daily`, `monthly`, or `full` | A clean run, a deliberate rerun, or an isolated failure lane  |
| Stakeholder                 | [Architecture](/docs/architecture)                        | Run `nbadb schema` for the fastest public-model snapshot                                                              | A clear picture of what the dataset contains and how it moves |

  Pick one lane, run one useful action, then branch. If you are not sure where
  to begin, choose the row that matches the job you need done **today**, not the
  title on your org chart. If you need the lowest-friction start possible, take
  the browser lane first and move to the local warehouse second.

## Pick your lane

Each tab gives you the same scan path:

1. **Run first** to get moving
2. **Open next** so you land on the right follow-up page
3. **Watch for** so the common misses do not cost a second possession

<Tabs items={["Contributor", "Analyst", "Operator", "Stakeholder"]}>
  <Tab value="Contributor">
    **Primary route:** [Installation](/docs/installation) → [CLI Reference](/docs/cli-reference) → [Architecture](/docs/architecture)

    ### Run first

    <CommandBlock label="Repo-local environment" command="uv sync --extra dev" />

    1. Keep the current command surface nearby in [CLI Reference](/docs/cli-reference).
    2. Read [Architecture](/docs/architecture) before changing pipeline flow, table families, or generated docs.

    ### Open next

    <div className="grid gap-4 md:grid-cols-2">
      <ScoutCard title="Schema Reference" label="Open next">
        Use <a href="/docs/schema">Schema Reference</a> to understand output naming and table families.
      </ScoutCard>
      <ScoutCard title="Data Lineage" label="Open next">
        Use <a href="/docs/lineage/table-lineage">Data Lineage</a> to trace a transform back upstream.
      </ScoutCard>
      <ScoutCard title="Regenerate docs" label="Open next">
        Run <code>uv run nbadb docs-autogen --docs-root docs/content/docs</code> after schema changes so generator-owned docs stay in sync.
      </ScoutCard>
    </div>

    ### Watch for

    <InsightCard title="Watch for">
      <ul>
        <li>Generator-owned docs should be regenerated, not hand-edited.</li>
        <li>Underscore-prefixed pipeline tables are internal operational state, not part of the public analytical contract.</li>
        <li>If CI or artifacts fail, jump to the <a href="/docs/guides/troubleshooting-playbook">Troubleshooting Playbook</a>.</li>
      </ul>
    </InsightCard>

  </Tab>

  <Tab value="Analyst">
    **Primary route:** [SQL Playground](/docs/playground) → [Analytics Quickstart](/docs/guides/analytics-quickstart) → [DuckDB Query Examples](/docs/guides/duckdb-queries)

    ### Run first

    <CommandBlock label="Seed a local dataset" command="nbadb download" />

    1. If you want a no-install warm-up first, open <a href="/docs/playground">SQL Playground</a> and run the leaderboard example before you download anything.
    2. Or use an existing <code>nba.duckdb</code> or Parquet export.
    3. Start with <code>analytics_*</code> surfaces for the fastest first query.
    4. Move into the table-level docs only after you know the question you want to answer.

    ### Open next

    <div className="grid gap-4 md:grid-cols-2">
      <ScoutCard title="SQL Playground" label="Open next">
        Use <a href="/docs/playground">SQL Playground</a> when you want one browser-only rehearsal before connecting to the real dataset.
      </ScoutCard>
      <ScoutCard title="DuckDB Query Examples" label="Open next">
        Use <a href="/docs/guides/duckdb-queries">DuckDB Query Examples</a> for a SQL-first scouting packet.
      </ScoutCard>
      <ScoutCard title="Player Comparison" label="Open next">

      </ScoutCard>

        Use <a href="/docs/guides/shot-chart-analysis">Shot Chart Analysis</a> for court-map and zone work.

    ### Watch for

    <InsightCard title="Watch for">
      <ul>
        <li>Keep <code>season_year</code> and <code>season_type</code> explicit so repeated analysis stays comparable.</li>
        <li>DuckDB is the fastest first possession for joins; Parquet is the cleaner handoff when the rest of the workflow is DataFrame-first.</li>
      </ul>
    </InsightCard>

  </Tab>

  <Tab value="Operator">
    **Primary route:** [Daily Updates](/docs/guides/daily-updates) → [CLI Reference](/docs/cli-reference) → [Troubleshooting Playbook](/docs/guides/troubleshooting-playbook)

    ### Run first

    <CommandBlock label="Check current pipeline state" command="nbadb status --output-format json" />

    1. If the state looks clean and you are on the recurring cadence, run <code>nbadb daily</code> next.
    2. Review what each run mode actually does: <code>daily</code>, <code>monthly</code>, <code>full</code>, <code>status</code>, <code>run-quality</code>.
    3. Confirm your data directory and output expectations.
    4. Keep the latest health artifacts in view:
       - <code>artifacts/health/daily/dependency-inventory.json</code>
       - <code>artifacts/health/daily/data-quality-report.json</code>
       - <code>artifacts/endpoint-coverage/*</code>

    ### Open next

    <div className="grid gap-4 md:grid-cols-2">
      <ScoutCard title="Troubleshooting Playbook" label="Open next">
        Use <a href="/docs/guides/troubleshooting-playbook">Troubleshooting Playbook</a> the moment a run leaves you with a failing artifact, a broken command, or a recovery question.
      </ScoutCard>
      <ScoutCard title="Pipeline Flow" label="Open next">
        Use <a href="/docs/diagrams/pipeline-flow">Pipeline Flow</a> to share a stage map quickly.

      <ScoutCard title="Kaggle Setup" label="Open next">
        Use <a href="/docs/guides/kaggle-setup">Kaggle Setup</a> for dataset handoff and publish flow.
      </ScoutCard>
    </div>

    ### Watch for

    <InsightCard title="Watch for">
      <ul>
        <li><code>daily</code>, <code>monthly</code>, and <code>full</code> end by rebuilding downstream tables in <code>replace</code> mode.</li>
        <li>A single <code>Ctrl+C</code> is the graceful stop; a second interrupt forces exit.</li>
        <li>When something breaks, start with the exact artifact or command named in the failure message.</li>
      </ul>
    </InsightCard>

  </Tab>
